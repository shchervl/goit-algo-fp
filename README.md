# goit-algo-fp

Фінальний проєкт курсу з алгоритмів та структур даних (GoIT × Neoversity).

## Налаштування

Проєкт використовує [uv](https://docs.astral.sh/uv/) для керування залежностями.


Це створить `.venv/`, встановить залежності та зареєструє виконуваний шорткат `pythagoras-tree`.

## Завдання 1. Структури даних. Сортування. Робота з однозв'язним списком

Реалізовано однозв'язний список з трьома операціями:

1. **Реверсування** — `LinkedList.reverse()`: змінює посилання між вузлами на місці, без створення нових вузлів.
2. **Сортування** — `LinkedList.sort()`: сортування злиттям (merge sort) за O(n log n).
3. **Об'єднання двох відсортованих списків** — `merge_sorted(a, b)`: повертає новий відсортований список без модифікації вхідних.

Файли: `task_01/linked_list.py`, тести в `task_01/tests/test_linked_list.py`.

### Запуск

```bash
# Запуск тестів (77 тестів)
uv run pytest task_01/

# Інтерактивне використання
uv run python -c "
from task_01.linked_list import LinkedList, merge_sorted

ll = LinkedList([3, 1, 4, 1, 5, 9, 2, 6])
ll.sort()
print(ll.to_list())  # [1, 1, 2, 3, 4, 5, 6, 9]

ll.reverse()
print(ll.to_list())  # [9, 6, 5, 4, 3, 2, 1, 1]

a = LinkedList([1, 3, 5])
b = LinkedList([2, 4, 6])
print(merge_sorted(a, b).to_list())  # [1, 2, 3, 4, 5, 6]
"
```

## Завдання 3. Дерева, алгоритм Дейкстри

Алгоритм Дейкстри для пошуку найкоротших шляхів у зваженому графі з використанням бінарної купи (`heapq`). Складність — O((V + E) log V).

Структура пакета `task_03/`:

| Модуль | Призначення |
|---|---|
| `graph.py` | `Graph[T]` — generic зважений граф (orієнтований/неорієнтований) на списку суміжності |
| `dijkstra.py` | `dijkstra(graph, start)` — мін-купа з lazy deletion; повертає `DijkstraResult[T]` (NamedTuple з `distances`, `predecessors`, `path_to()`) |
| `demo.py` | Класичний приклад: 6 вершин, 9 ребер, друк відстаней та шляхів від `A` |
| `tests/` | 62 тести: граф, Дейкстра, реконструкція шляху, цикли, `None`, граничні випадки |

Реалізація використовує **lazy deletion** замість `decrease-key`: при знаходженні кращої ваги нова пара `(dist, node)` штовхається у купу; старі (stale) пари пропускаються при витяганні через перевірку `dist > distances[u]`.

### Запуск

```bash
uv run python -m task_03.demo
```

Очікуваний вивід (класичний приклад взятий з Вікіпедії):

```
Куди   Відстань   Шлях
----------------------------------------
A      0.0        A
B      7.0        A → B
C      9.0        A → C
D      20.0       A → C → D
E      20.0       A → C → F → E
F      11.0       A → C → F
```

### Використання в коді

```python
from task_03.graph import Graph
from task_03.dijkstra import dijkstra

g: Graph[str] = Graph(directed=False)
g.add_edge("A", "B", 7)
g.add_edge("A", "C", 9)
g.add_edge("B", "C", 10)

result = dijkstra(g, start="A")
print(result.distances["C"])    # 9.0
print(result.path_to("C"))      # ['A', 'C']

# Tuple-unpacking теж працює (для зворотньої сумісності):
distances, predecessors = dijkstra(g, "A")
```

## Завдання 6. Жадібні алгоритми та динамічне програмування

Класична задача 0/1-ранця: вибір страв з обмеженим бюджетом для максимізації калорій. Реалізовано **два підходи** для порівняння:

- `greedy_algorithm(items, budget)` — жадібний: сортує за калорії/вартість, бере доки бюджет дозволяє. O(n log n). Швидкий, але **не гарантує оптимуму** для 0/1-ранця.
- `dynamic_programming(items, budget)` — DP-таблиця `dp[i][b]` з backtracking. O(n·B). **Завжди оптимальний** результат.

Обидві функції повертають `Selection(items, total_cost, total_calories)` — NamedTuple з підтримкою tuple-unpacking і attribute-access.

Структура пакета `task_06/`:

| Модуль | Призначення |
|---|---|
| `knapsack.py` | `greedy_algorithm`, `dynamic_programming`, `Selection` |
| `demo.py` | Порівняння на бюджетах 50/80/100/150 |
| `tests/` | 19 тестів: коректність, інваріант DP ≥ greedy, граничні випадки |

### Запуск

```bash
uv run python -m task_06.demo
```

При бюджеті **100** DP знаходить `pizza+cola+potato+pepsi = 970` кал (вписався точно), а greedy зупиняється на `cola+potato+pepsi+hot-dog = 870` кал (лишає 20 неспожитих). Це канонічний приклад **суб-оптимальності жадібного алгоритму** для 0/1-ранця.

## Завдання 7. Метод Монте-Карло

Симуляція кидків двох кубиків (1–6) з оцінкою емпіричних ймовірностей кожної суми 2–12 та порівнянням до аналітичних значень `P(s) = (6 − |s − 7|) / 36`.

Структура пакета `task_07/`:

| Модуль | Призначення |
|---|---|
| `monte_carlo.py` | `simulate(n_rolls, seed)`, `empirical_probabilities(counts)`, `analytical_probabilities()`, `comparison_table()`, `plot_comparison()` |
| `demo.py` | 100 000 кидків + таблиця + графік порівняння |
| `tests/` | 18 тестів: коректність діапазону, відтворюваність seed, нормалізація, **закон великих чисел** (відхилення < 1% при N=200к) |

### Запуск

```bash
uv run python -m task_07.demo
```

| 12 | 2.81% | 2.78% (1/36) | 2 783 | +0.04% |

### Висновки

**Метод Монте-Карло коректно відтворює аналітичні ймовірності.** За 100 000 кидків:

1. **Збіг із теорією**: всі емпіричні ймовірності відхиляються від теоретичних не більше ніж на **0.21%**. Це повністю узгоджується з очікуваною похибкою порядку `O(1/√N) ≈ 0.32%` для `N = 10⁵`.

2. **Підтверджено форму розподілу**: симетричний трикутник з піком на сумі 7 (`P = 6/36 ≈ 16.67%`) — це наслідок того, що `7 = 1+6 = 2+5 = 3+4 = 4+3 = 5+2 = 6+1` (6 способів з 36), тоді як крайні значення 2 і 12 досягаються лише однією комбінацією кожне (`1+1` та `6+6`).

3. **Закон великих чисел у дії**: при збільшенні `N` від 1 000 до 200 000 максимальне відхилення монотонно зменшується (тест `test_converges_to_analytical` гарантує < 1% при N = 200 000). Це експериментальне підтвердження теореми про збіжність відносної частоти до ймовірності.

4. **Висновок про метод**: Монте-Карло — валідний інструмент для оцінки ймовірностей дискретних розподілів. Для досягнення точності 0.1% достатньо ~10⁶ випробувань; для 1% — ~10⁴. У цьому випадку аналітичний підхід **точніший** (видає точні дроби), але Монте-Карло універсальний — працює там, де аналітика неможлива або надто складна.
